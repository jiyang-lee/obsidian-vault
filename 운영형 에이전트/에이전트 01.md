# 1. 먼저 가장 중요한 결론


```
센서 데이터
   ↓
머신러닝 모델이 위험도 계산
   ↓
우선순위 카드 생성
   ↓
데이터베이스 저장
   ↓
위험한 카드가 알림으로 등록
   ↓
운영자가 화면에서 알림 선택
   ↓
백엔드 서버가 에이전트 실행
   ↓
에이전트가 DB·RAG·날씨·모델 결과 확인
   ↓
LLM 또는 규칙 기반 fallback이 설명과 조치 작성
   ↓
운영자가 검토
   ↓
보고서·작업지시서로 저장
```

>머신러닝 모델은 위험을 계산하고, 에이전트는 그 위험을 조사하고 설명

에이전트는 만들어진 priority card 에서 시작한다.

# 2. 등장인물

## 프론트엔드

운영자가 보는 화면이다.

```
대시보드 | 알림 | AI 활동 | 설정 | 관리자
```

프론트엔드는 서버에게 요청하고 결과를 보여준다.
예를 들어, 운영자가 알림 화면에서 작업지시서 생성을 누르면 
프론트는 서버에 이렇게 요청한다.

```
이 알림을 가지고 에이전트를 실행해 주세요.
```

현재 프론트엔드는 React, TypeScript, Vite로 구성되어 있다.

---
## 백엔드 서버

프론트엔드의 요청을 받는 프로그램이다.

### 서버 프로그램

이 프로젝트의 서버 프로그램은 FastAPI로 만들어졌고, 포트는 8002이다.
프론트엔드는 기본적으로 5173 포트에서 실행된다.

서버가 받는 요청:
```
알림 목록 주세요.
알림을 확인 처리해 주세요.
에이전트를 실행해 주세요.
에이전트 결과를 주세요.
보고서를 저장해 주세요.
```
---
## 데이터베이스

프로젝트가 기억해야 할 정보를 저장하는 곳, PostgreSQL 사용

데이터베이스 정보:
```
센서 윈도우
모델 판단 결과
우선순위 카드
알림
에이전트 실행 기록
에이전트 단계 기록
보고서
작업지시서
운영자 검토 기록
```

데이터베이스는 기억 창고이다.
서버가 꺼졌다 켜져도 결과가 남아 있으려면 데이터베이스에 저장해야 한다.
변수에만 넣어두면 서버가 죽는 순간 사라진다.

---
## 머신러닝 모델

머신러닝 모델은 센서 데이터를 보고 점수를 계산한다.

```
이상 징후
위험 확률
고장까지 남은 예상 시간
M1 전용 판단
최종 우선순위
```

만들어내는 결과:
```
31번 기계실

위험도: 높음
이상 징후: 감지
예상 대응 시간: 24시간 이내
우선순위: 긴급
```

이 결과를 Priority Card, 우선순위 카드로 저장한다.

---
## 알림

알림은 위험한 우선순위 카드를 운영자가 처리할 수 있도록 큐에 등록한 것.

```
Priority Card
   ↓
긴급 또는 높은 위험인가?
   ↓
맞으면 알림 생성
```

알림 속 정보:
```
어느 기계실인가
위험 등급은 무엇인가
우선순위 점수는 얼마인가
현재 상태는 미확인·확인·종결 중 무엇인가
```

---
## 에이전트

에이전트는 알림 하나를 받아 조사하고 결과를 정리하는 실행 흐름이다.

```
필요한 데이터 가져오기
근거 확인하기
모델 결과 다시 확인하기
근거가 부족한지 판단하기
필요하면 추가 조사하기
답변 작성하기
답변 검증하기
운영자 검토 요청하기
결과 저장하기
```
---
## LLM

LLM은 에이전트 안에서 문장을 만들고 판단을 보조하는 구성 요소
```
에이전트 = 전체 업무 담당자
LLM = 담당자가 사용하는 사고·문장 작성 도구
```

> 에이전트가 필요할 때 LLM을 호출한다.

---
## RAG

rag는 에이전트가 참고할 수 있는 내부 자료 검색 기능이다.

포함 문서:
```
설비 운영 매뉴얼
고장 대응 가이드
과거 점검 기록
지역난방 운영 기준
```

|구분|역할|
|---|---|
|PostgreSQL|정해진 구조의 운영 데이터 저장|
|RAG|문서에서 관련 내용을 검색|
|LLM|데이터를 읽고 설명 작성|
|머신러닝 모델|위험 점수 계산|

# 2. 전체 시스템의 큰 단계: 6단계

## 1단계. 센서 데이터 준비

```
원시 센서 데이터
→ 일정 시간 단위로 묶기
→ 통계 특징 만들기
```

[[6시간 윈도우]] = 모델 입력값:
```
평균
최댓값
최솟값
변화량
결측 비율
```

---
## 2단계. 머신러닝 모델 판단

```
센서 특징
→ 이상 탐지
→ 위험도 예측
→ 대응 시간 예측
→ 우선순위 계산
```

결과는 우선순위 카드로 만들어진다.
```
card_id
기계실 번호
위험 점수
우선순위 등급
판단 이유
권장 조치
검토 필요 여부
```

---
## 3단계. 데이터베이스와 알림

우선순위 카드는 PostgreSQL에 저장된다.
그 중 위험도가 높은 카드는 알림으로 등록된다.

```
전체 Priority Card
   ↓
urgent / high만 선택
   ↓
Alert Queue 저장
```
---
## 4단계. 운영자가 에이전트 실행

운영자가 알림 화면에서 버튼을 누른다. -> 프론트는 다음 API를 호출한다.

```http
POST /api/agent-runs
```

내용:
```
{
  "alert_id": "현재 선택한 알림",
  "requested_by": "ops-manager",
  "reason": "운영 콘솔에서 분석 요청"
}
```

서버는 새로운 run_id를 만든다.
```
alert_id = 어떤 문제를 처리하는가
run_id = 이번 에이전트 실행 번호
```

같은 알림이라도 에이전트를 여러 번 실행할 수 있으므로 둘을 구분한다.

---
## 5단계. 에이전트 조사와 답변 생성

에이전트 순서:
```
실행 시작 표시
→ 운영 입력 불러오기
→ DB 근거 가져오기
→ RAG·날씨 정보 가져오기
→ 모델 결과 재검증
→ 근거 충분성 평가
→ 필요하면 추가 조사
→ 답변 생성
→ 답변 검증
```
현재 코드의 실제 LangGraph 흐름도 이 구조로 구현되어 있다.

근거가 부족하면 에이전트는 다음 중 하나를 선택한다.
```
내부 근거 더 찾기
모델 결과 다시 검증하기
진단 Worker 실행하기
현재 근거로 최종 답변 작성하기
```
---
## 6단계. 운영자 검토와 결과 저장

에이전트가 결과를 만들었다고 바로 정비업체에 보내지는 않는다.
```
에이전트 결과
→ 운영자 검토
→ 승인·수정·보류
→ 보고서 또는 작업지시서 저장
```

이 시스템은 의사결정 보조 시스템이므로 최종 판단자는 사람이다.

# 4. server.py 는 무슨 일을 하는가

server.py 는 백엔드의 입구

서버가 켜질 때 다음 작업을 한다.
```
1. 데이터베이스 연결
2. 필요한 테이블 확인
3. PostgreSQL connection pool 생성
4. LangGraph checkpoint 준비
5. 에이전트 그래프 생성
6. 중단된 에이전트 실행 복구
7. API 요청 대기
```

쉽게:
```
server.py = 건물 정문 + 안내 데스크 + 개점 준비 담당자
```

그러나 모든 일을 직접 하지는 않는다.
```
server.py
→ router에게 API 처리 위임
→ repository에게 DB 조회 위임
→ agent graph에게 에이전트 실행 위임
```

# 5. 에이전트의 구성

크게 다섯 부분이다.

## 1. State

>에이전트가 현재 무엇을 알고 있는지 저장한다.

```
어떤 알림을 처리 중인가
어떤 근거를 찾았는가
몇 번째 반복인가
모델 재검증 결과는 무엇인가
최종 답변은 만들어졌는가

src/heatgrid_ops/agent/state.py

state는 에이전트의 작업 메모장이다.
```
---
## 2. Node

> 각각의 실제 작업

```
운영 데이터 불러오기
근거 찾기
날씨 찾기
모델 검증하기
진단 Worker 실행하기
답변 생성하기
답변 검증하기

nodes_input.py
nodes_evidence.py
nodes_diagnostic.py
nodes_output.py
nodes_completion.py

Node 하나가 업무 하나.
```
---
## 3. Graph

> Node를 어떤 순서로 실행할지 정한다.

```
근거가 충분함
→ 답변 생성

근거가 부족함
→ 근거 추가 검색
→ 다시 판단

모델 결과가 의심스러움
→ 모델 재검증
→ 다시 판단

src/heatgrid_ops/agent/graph.py

LangGraph는 조건에 따라 갈라지고 반복되는 업무 흐름이다.
```
---
## 4. Port와  Adapter

> 에이전트 핵심 코드는 외부 기술을 직접 알지 않도록 만들어져 있다.

```
에이전트: 데이터를 가져와 줘.
Adapter: PostgreSQL에서 가져올게.

에이전트: 문서를 찾아 줘.
Adapter: pgvector로 검색할게.

에이전트: 답변을 만들어 줘.
Adapter: OpenAI 모델을 호출할게.
```
---
## 5. Checkpoint와 Audit

> 에이전트 실행 과정을 저장한다.

```
어디까지 실행했는가
어떤 결정을 했는가
어떤 도구를 사용했는가
토큰은 얼마나 썼는가
실패하면 어디서 다시 시작할 것인가

PostgreSQL 기반 LangGraph checkpoint를 사용한다.
각 실행의 run_id가 checkpoint의 thread_id로 사용된다.
```

# 6. 하나의 예시

31번 기계실에 누수 위험이 있다고 가정한다.

#### 모델 단계
```
센서 압력 감소
보충수 유량 증가
온도 패턴 이상

risk_score = 0.91
priority_level = urgent
leadtime = 0~24시간
```

모델이 판단하고 우선순위 카드가 DB에 저장된다.

---
#### 알림 단계
높은 위험이므로 알림이 만들어진다.
```
31번 기계실
누수 위험 의심
긴급
미확인
```
---
#### 운영자 단계
운영자가 알림을 클릭하고 AI 분석 시작을 누른다.
```
POST /api/agent-runs
```
---
#### 에이전트 단계
에이전트가 조사한다.
```
우선순위 카드 조회
센서 요약 조회
모델 결과 조회
RAG에서 누수 대응 매뉴얼 검색
현재 날씨와 부하 확인
모델 결과 재검증
```
그리고 판단한다.
```
근거가 충분한가?
모델 간 결과가 일치하는가?
사람 검토가 필요한가?
```
----
#### 결과 단계
에이전트가 초안을 만든다.
```
상황:
31번 기계실에서 압력 감소와 보충수 유량 증가가 함께 나타났습니다.

권장 조치:
1. 현장 누수 여부 확인
2. 보충수 계통 점검
3. 차압 센서 상태 확인

주의:
모델 결과는 누수를 확정하지 않으며 현장 확인이 필요합니다.
```
---
#### 운영자 검토 단계
운영자는 결과를 다음 중 하나로 처리한다.
```
승인
수정
사람 검토 유지
```
승인된 결과가 보고서 또는 작업지시서가 된다.
